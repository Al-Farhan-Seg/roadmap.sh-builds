## About This Repository

This repository documents our progress through the roadmap.sh frontend project collection. Each project is kept in a numbered folder, following a consistent structure such as:

```text
7-accessible-form-ui/
├── index.html
├── styles.css
├── README.md
└── .project-complete   # Added only when the project is genuinely finished
```

Vanilla JavaScript projects include `script.js` where interactive behaviour is required. Framework-based projects may later use their framework's recommended structure instead.

Through these projects, we will practise:

- semantic and accessible HTML;
- responsive CSS layouts using Flexbox and Grid;
- DOM manipulation and browser-side JavaScript;
- API integration and asynchronous programming;
- reusable components and frontend frameworks;
- clean project organisation and technical documentation.

## Status System

| Status | Meaning |
|:---:|---|
| ✅ | The project folder contains `.project-complete` and is counted in the completion percentage. |
| 🚧 | The folder and starter files exist, but the project is still in progress. |
| ⬜ | The project folder has not been created yet. |

This prevents generated starter files from being mistaken for finished projects.

## Repository Automation

The repository includes the following automation files:

```text
scripts/
├── project-status.mjs
├── projects-fallback.json
├── scaffold-projects.mjs
└── update-readme.mjs
```

### `scaffold-projects.mjs`

Creates missing numbered project folders and their starter files without overwriting existing work.

```bash
node scripts/scaffold-projects.mjs
```

To scaffold projects starting from a specific project number:

```bash
node scripts/scaffold-projects.mjs --from=8
```

To preview the operation without creating files:

```shell
node scripts/scaffold-projects.mjs --from=8 --dry-run
```

### `project-status.mjs`

Adds or removes the `.project-complete` marker.

```shell
# Mark project 8 as complete
node scripts/project-status.mjs complete 8

# Mark several projects complete
node scripts/project-status.mjs complete 1-7

# Return project 8 to in-progress status
node scripts/project-status.mjs in-progress 8

# Inspect all project statuses
node scripts/project-status.mjs status 1-30
```

### `projects-fallback.json`

Stores the trusted local roadmap.sh project catalogue used by the repository scripts.

The catalogue is updated manually after reviewing changes to the official roadmap.sh frontend project list. This prevents incomplete external responses from unexpectedly removing projects from the tracker.
### `update-readme.mjs`

Reads the trusted local project catalogue, scans numbered project folders, calculates progress, and regenerates only the project tracker between the README markers.

```bash
node scripts/update-readme.mjs
```

Run this script locally after changing a project's status. Review the generated changes with `git diff` before committing and pushing.

## Running a Project Locally

Clone the repository:

```bash
git clone https://github.com/Al-Farhan-Seg/roadmap.sh-builds.git
cd roadmap.sh-builds
```

Open a project folder and launch its `index.html` file in a browser. A development extension such as **Live Server** can automatically refresh the page while changes are being made.

Projects created with a framework may later replace the basic starter files with the framework's recommended structure and development commands.

## Project Standards

Each completed project should aim to include:

- semantic HTML and meaningful document structure;
- accessible names, labels, focus states, and keyboard support;
- responsive behaviour across mobile and desktop screens;
- organised and consistently formatted code;
- a project-specific `README.md` linked to the official brief;
- screenshots or a live demonstration where appropriate;
- a `.project-complete` marker only after the implementation has been reviewed.
